Часть 1: Fat JAR — Упаковка приложения

Что такое Fat JAR (Uber JAR)?

Простое объяснение: Fat JAR — это один файл, содержащий ваше приложение и все его зависимости. Как коробка с пиццей — внутри уже всё, что нужно для употребления. 
┌─────────────────────────────────────────────────────────────────────────────┐
│                           Обычный JAR vs Fat JAR                            │
└─────────────────────────────────────────────────────────────────────────────┘

     ОБЫЧНЫЙ JAR                              FAT JAR (Executable)
     ───────────                              ────────────────────
    ┌───────────────┐                        ┌───────────────────────────────┐
    │  Ваш код      │                        │  Ваш код                      │
    │  (.class)     │                        │  (.class файлы)               │
    └───────────────┘                        ├───────────────────────────────┤
          │                                  │  BOOT-INF/lib/                │
          │ Нужны внешние                    │  ├── spring-boot-3.5.5.jar    │
          │ библиотеки!                      │  ├── spring-web-6.2.10.jar    │
          ▼                                  │  ├── hibernate-6.6.26.jar     │
    ┌───────────────┐                        │  ├── postgresql-42.7.x.jar    │
    │ spring-boot   │                        │  └── ... (100+ зависимостей)  │
    │ hibernate     │                        ├───────────────────────────────┤
    │ postgresql    │                        │  META-INF/MANIFEST.MF         │
    │ ...           │                        │  Main-Class: JarLauncher      │
    └───────────────┘                        └───────────────────────────────┘
    
    ❌ Нужно настраивать                      ✅ Просто запустить:
       classpath                                java -jar app.jar

Зачем нужен Fat JAR?

Преимущество Описание
Простота развёртывания Один файл — скопировал и запустил. Не нужно устанавливать зависимости.
Воспроизводимость Версии всех библиотек зафиксированы внутри JAR.
Портативность Работает везде, где есть Java. Windows, Linux, macOS — без разницы.
Основа для Docker Fat JAR легко упаковать в Docker-контейнер.

Как собрать Fat JAR

1 Добавить Spring Boot Maven Plugin в pom.xml
pom.xml
<build>
    <plugins>
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
            <configuration>
                <excludes>
                    <exclude>
                        <groupId>org.projectlombok</groupId>
                        <artifactId>lombok</artifactId>
                    </exclude>
                </excludes>
            </configuration>
        </plugin>
    </plugins>
</build>

Lombok исключаем — он нужен только при компиляции, не в runtime.

2 Собрать проект 
mvn clean package -DskipTests
3 Запустить 
java -jar target/Android-Bootcamp-2025-Backend-Liquibase-1.0-SNAPSHOT.jar

Часть 2: Docker — Контейнеризация приложения

Что такое Docker?

Простое объяснение: Docker — это способ упаковать приложение вместе с его окружением (ОС, Java, настройки) в изолированный контейнер. Контейнер работает одинаково на любом компьютере.

📦 Аналогия: Контейнеровоз

                    МОРСКИЕ ПЕРЕВОЗКИ

    Раньше (без контейнеров):          Сейчас (с контейнерами):
    
    🚢 Корабль                          🚢 Корабль
    ┌─────────────────────┐             ┌─────────────────────┐
    │ 🍎🥕🧱📺🛢️🪑        │             │ ┌───┐ ┌───┐ ┌───┐  │
    │   🧀 📦 🎸 🛞       │             │ │ 📦│ │ 📦│ │ 📦│  │
    │  Всё вперемешку!    │             │ └───┘ └───┘ └───┘  │
    │  Долгая погрузка    │             │ Стандартные размеры │
    └─────────────────────┘             └─────────────────────┘
    
    ❌ Хрупкое рядом с тяжёлым          ✅ Изолированно
    ❌ Каждый порт — свои правила       ✅ Унифицировано
    ❌ Сложная логистика                ✅ Автоматизация

Docker-контейнер = стандартный "ящик" для программ. Работает везде одинаково!

Dockerfile — рецепт сборки образа

Dockerfile — это текстовый файл с инструкциями, как построить Docker-образ. Образ — это шаблон, из которого создаются контейнеры.
Dockerfile
# ==================== ЭТАП 1: СБОРКА ====================
FROM maven:3.9-eclipse-temurin-17-alpine AS build
WORKDIR /app
COPY pom.xml .
COPY src ./src
RUN mvn clean package -DskipTests

# ==================== ЭТАП 2: ЗАПУСК ====================
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

Разбор Dockerfile по шагам

┌─────────────────────────────────────────────────────────────────────────────┐
│                        Multi-stage Build (Многоэтапная сборка)              │
└─────────────────────────────────────────────────────────────────────────────┘

   ЭТАП 1: build                              ЭТАП 2: runtime
   (Сборочный контейнер)                      (Финальный образ)
   
  ┌────────────────────────┐                ┌────────────────────────┐
  │ maven:3.9-temurin-17   │                │ eclipse-temurin:17-jre │
  │ (~400 MB)              │                │ (~150 MB)              │
  ├────────────────────────┤                ├────────────────────────┤
  │                        │                │                        │
  │  COPY pom.xml          │                │  COPY app.jar          │
  │  COPY src/             │   ─────────▶   │  (из этапа build)      │
  │  RUN mvn package       │   только JAR   │                        │
  │                        │                │  ENTRYPOINT java -jar  │
  │  Результат: app.jar    │                │                        │
  └────────────────────────┘                └────────────────────────┘
  
        ❌ Отбрасывается                          ✅ Используется
        (Maven, исходники)                        (только JRE + JAR)
Зачем multi-stage? Финальный образ содержит только то, что нужно для запуска. Нет Maven, нет исходников, нет компилятора — меньше размер, меньше уязвимостей.
Инструкция Что делает
FROM Базовый образ (на чём строим)
WORKDIR Рабочая директория внутри контейнера
COPY Копировать файлы в образ
RUN Выполнить команду при сборке
EXPOSE Документировать порт (не открывает!)
ENTRYPOINT Команда при запуске контейнера

Команды Docker

# Собрать образ
docker build -t bootcamp-app .

# Посмотреть образы
docker images

# Запустить контейнер
docker run -p 8080:8080 bootcamp-app

# Посмотреть запущенные контейнеры
docker ps

# Остановить контейнер
docker stop <container_id>

Часть 3: Docker Compose — Оркестрация контейнеров

Зачем нужен Docker Compose?

Реальные приложения состоят из нескольких сервисов: backend, база данных, кэш, очередь сообщений... Docker Compose позволяет описать их все в одном файле и запустить одной командой. 
┌─────────────────────────────────────────────────────────────────────────────┐
│                              Без Docker Compose                             │
└─────────────────────────────────────────────────────────────────────────────┘

  Терминал 1:                    Терминал 2:                    Терминал 3:
  ┌──────────────────┐          ┌──────────────────┐          ┌──────────────────┐
  │ docker run       │          │ docker run       │          │ docker run       │
  │   --name pg      │          │   --name app     │          │   --name redis   │
  │   -e POSTGRES_.. │          │   -p 8080:8080   │          │   -p 6379:6379   │
  │   -p 5432:5432   │          │   --link pg      │          │   redis:alpine   │
  │   postgres:16    │          │   bootcamp-app   │          │                  │
  └──────────────────┘          └──────────────────┘          └──────────────────┘
  
  ❌ Много команд
  ❌ Нужно помнить порядок запуска
  ❌ Сложно остановить всё
  ❌ Нет автоматического рестарта


┌─────────────────────────────────────────────────────────────────────────────┐
│                              С Docker Compose                               │
└─────────────────────────────────────────────────────────────────────────────┘

  Терминал:
  ┌───────────────────────────────────────────────────────────────────────────┐
  │                                                                           │
  │  $ docker-compose up                                                      │
  │                                                                           │
  │  ✅ PostgreSQL started                                                    │
  │  ✅ Redis started                                                         │
  │  ✅ App started (waited for dependencies)                                 │
  │                                                                           │
  └───────────────────────────────────────────────────────────────────────────┘
  
  ✅ Одна команда
  ✅ Автоматический порядок запуска
  ✅ docker-compose down — остановить всё
  ✅ Перезапуск при падении

Структура docker-compose.yml

docker-compose.yml
version: '3.8'

services:
  # ==================== БАЗА ДАННЫХ ====================
  postgres:
    image: postgres:16-alpine          # Готовый образ из Docker Hub
    container_name: bootcamp-postgres
    environment:                        # Переменные окружения
      POSTGRES_DB: bootcamp_db
      POSTGRES_USER: bootcamp
      POSTGRES_PASSWORD: bootcamp123
    ports:
      - "5432:5432"                     # host:container
    volumes:
      - postgres_data:/var/lib/postgresql/data   # Сохранение данных
    healthcheck:                        # Проверка готовности
      test: ["CMD-SHELL", "pg_isready -U bootcamp -d bootcamp_db"]
      interval: 10s
      timeout: 5s
      retries: 5

  # ==================== ПРИЛОЖЕНИЕ ====================
  app:
    build:                              # Собрать из Dockerfile
      context: .
      dockerfile: Dockerfile
    container_name: bootcamp-app
    ports:
      - "8080:8080"
    environment:
      SPRING_PROFILES_ACTIVE: docker
      SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/bootcamp_db
      SPRING_DATASOURCE_USERNAME: bootcamp
      SPRING_DATASOURCE_PASSWORD: bootcamp123
    depends_on:                         # Зависимость от postgres
      postgres:
        condition: service_healthy      # Ждать пока БД готова

# ==================== ТОМА ДЛЯ ДАННЫХ ====================
volumes:
  postgres_data:                        # Именованный том

Ключевые концепции

Концепция Описание Пример
services Контейнеры для запуска postgres, app
image Готовый образ из Docker Hub postgres:16-alpine
build Собрать образ из Dockerfile context: .
ports Проброс портов (host:container) "8080:8080"
environment Переменные окружения POSTGRES_USER: bootcamp
volumes Постоянное хранилище postgres_data:/var/lib/...
depends_on Порядок запуска app зависит от postgres
healthcheck Проверка готовности сервиса pg_isready

Команды Docker Compose

# Собрать и запустить все сервисы
docker-compose up --build

# Запустить в фоновом режиме
docker-compose up -d

# Посмотреть логи
docker-compose logs -f

# Посмотреть логи конкретного сервиса
docker-compose logs -f app

# Остановить все контейнеры
docker-compose down

# Остановить и удалить данные (volumes)
docker-compose down -v

# Перезапустить один сервис
docker-compose restart app

# Пересобрать без кэша
docker-compose build --no-cache
